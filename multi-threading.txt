- Thread T1 is currently executing the static method updateInterestRate().
  At the exact same time, Thread T2 tries to execute the instance method withdrawMoney() on a specific BankAccount object.
  Does T2 have to wait for T1?

  The Answer: NO. They run perfectly in parallel.

  Why? Because they are grabbing two completely separate locks.

  Thread T1 has the key to the Master Class Safe (BankAccount.class).

  Thread T2 has the key to the Individual Object Safe (this).

  The locks are entirely independent of each other.

- What is Happens before guarantee?? what is instruction reordering??


  Our CPU and java compiler is obsessed with making process fast. If the compiler or the CPU looks at your code and realizes that changing the order of the
  instructions will make it run faster without changing the final result, it will rearrange your code.

  public class ReorderingProblem {
    int data = 0;
    boolean isReady = false;

    // Thread 1 executes this
    public void prepareData() {
        data = 42;          // Line A
        isReady = true;     // Line B
    }

    // Thread 2 executes this
    public void processData() {
        if (isReady) {             // Line C
            System.out.println(data); // Line D
        }
    }
}


  problem occurs here:

  Look closely at Thread 1. To the CPU, data = 42 and isReady = true have absolutely nothing to do with each other. They don't depend on each other.
  So, to optimize performance, the CPU might execute Line B before Line A.

  If Thread 2 happens to run exactly at that split second, it sees isReady is true, enters the if statement, and prints data... which is still 0!

  This is an impossible-to-reproduce bug caused entirely by hardware optimization.


- Here is exactly how synchronized stops the chaos:

  
1) It stops Reordering: The compiler is not allowed to take data = 42 and move it outside of the synchronized block. The walls of the synchronized block are
   impenetrable. Whatever happens inside, stays inside, and executes in a safe, isolated bubble.



Question) Can overriden methods like run cannot throws InterruptedException

Java has a strict inheritance rule: An overriding method cannot throw any new or broader checked exceptions than the overridden method.
Because the original run() method doesn't throw anything, your run() method isn't allowed to either.

+ 


Why didn't the Java architects just add throws Exception to the Runnable interface?

Think about how standard methods work. If Method A calls Method B, and Method B throws an error, Method B passes that error back down the call stack to Method A
 to handle.

But threads don't share a call stack. Threads are entirely independent timelines.

Your main thread calls t1.start().

The OS creates a brand new thread (t1) with its own isolated stack.

The main thread instantly moves on to the next line of code. It does not wait around.

Three seconds later, t1 is executing run() and encounters an InterruptedException.

If Java allowed t1 to "throw" that exception out of the run() method... who is there to catch it? The main thread is already gone. There is no "parent" waiting to
catch the thrown exception.

- the difference between run and start method boils down to the concept of stack (container) and stack frame (container content)
- in java if you have main thread and 3 other threads you have basically 4 stacks
- each stack is private
- A Stack Frame is a data structure that gets created every single time you call a method. It lives inside the Stack.
- when you call run() - the stack frame is added to main thread stack due to which main thread gets blocked
- but when you call start() - a new thread is created and a new stack is created and the stack frame is pushed into that thread stack not on main thread stack

package com.springboot.tutorial;

import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.BlockingQueue;
import java.util.concurrent.LinkedBlockingQueue;

class Worker implements Runnable
{
    private BlockingQueue<Runnable> queue;
    private Thread currentThread;
    private volatile boolean isStopped = false;
    private int threadId;

    Worker(BlockingQueue<Runnable> queue, int id){
        this.queue=queue;
        this.threadId = id;
    }

    @Override
    public void run(){
        currentThread = Thread.currentThread();
        Thread.currentThread().setName("Thread Name "+threadId);
        System.out.println("Thread "+threadId+" started");
        while(!isStopped){
            Runnable task = null;
            try {
                task = this.queue.take();
                task.run();
            } catch (InterruptedException e) {
                System.out.println("Thread "+threadId+" ended");
                break;
            }
        }
    }
    public void stopWorker(){
        this.isStopped=true;
        this.currentThread.interrupt();
    }
}

class ThreadPoolManager{

    private BlockingQueue<Runnable> queue;
    private List<Worker> workers = new ArrayList<>();
    private boolean stop = false;

    ThreadPoolManager(int threads){

        this.queue = new LinkedBlockingQueue<>();

        for(int i=0;i<3;i++){
            Worker worker = new Worker(queue,i+1);
            workers.add(worker);

            Thread t1 = new Thread(worker);
            t1.start();
        }
    }

    public synchronized void startProcess(Runnable task){
        if(!stop) this.queue.offer(task); // difference between add and offer
    }

    public synchronized void stopWorker(){
        if(!stop){
            for(Worker worker: workers){
                worker.stopWorker();
            }
        }
    }
}

public class ThreadPoolCustom {
    public static void main(String[] args){

        ThreadPoolManager manager = new ThreadPoolManager(3);


        Thread t1 = new Thread(()->{
            for(int i=0;i<=100;i++){
                int finalI = i;
                manager.startProcess(()->{
                    System.out.println("Task Number "+ finalI + " completed by "+ Thread.currentThread().getName());
                });
            }
        });

        t1.start();
    }

}



** JAVA LOCKS **

- Reentrance means that if a thread already holds a lock, it is allowed to "re-enter" any other block of code protected by that exact same lock instance
  without getting stuck.

- class LockSimulation implements Runnable{

    public static int count=0;
    private final Lock lock = new ReentrantLock(true);

    public void kuchKaam(){
        lock.lock(); // this lock instace here is acquired by thread A that has called run, hence it is allowed to lock again
        try {
            System.out.println("Main hoon inside kuch kaam "+Thread.currentThread().getName());
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        finally {
            lock.unlock();
        }
    }
    @Override
    public void run() {
        lock.lock(); // thread A has acquired lock on this instance
        System.out.println("Maine lagaya lock " + Thread.currentThread().getName());
        count++;
        try {
            Thread.sleep(2000);
            kuchKaam();
        } catch (InterruptedException ex) {
            System.out.println("Error occured");
        } finally {
            lock.unlock();
        }
    }
}

- The standard lock.lock() and the synchronized keyword are uninterruptible. If a thread is waiting for a synchronized block, you cannot stop it.
  Even if you call thread.interrupt(), the thread stays stuck in the "Blocked" state until it gets the lock.
 
- tryLock() is the most popular tool for avoiding Deadlocks. It asks the question: "Is the lock available right now?"

- public void performTask() {
    if (lock.tryLock()) { // "Hey, can I have the key?"
        try {
            // Critical Section: I have the lock!
            System.out.println("Lock acquired, doing work...");
        } finally {
            lock.unlock();
        }
    } else {
        // Plan B: I didn't get the lock.
        System.out.println("Lock busy. I'll try again later or do something else.");
    }
}

- tryLock() - This person walks up to the door, tries the handle, finds it's locked, and says: "Forget it, I have work to do."

  They do not wait.

  They immediately go back to their desk.

  This is used when you have a "Plan B."

- lock.lockInterruptibly() (The Responsive Employee)

  This person stands outside the door to wait, but they keep their phone in their hand.

  They are willing to wait, BUT if their boss calls and says "Come to my office now!" (Interruption), they leave the line immediately.

  Unlike the stubborn colleague, they are responsive to the outside world while waiting.

- In your ThreadPoolManager, when you call stopPool(), you want your workers to die. If a worker is stuck waiting for a lock.lock(), it will ignore your 
  shutdown command. It will stay alive until it gets that lock.

- When you call targetThread.interrupt(), two things happen depending on what the target is doing:

1) Case A: The Thread is "Waiting" (Sleep, Wait, Join, Take)

   If the thread is currently inside a method like Thread.sleep(), Object.wait(), or BlockingQueue.take(), the JVM does the following:

   It clears the "interrupted status" flag.

   It throws an InterruptedException inside the target thread's stack.

   This "shove" forces the thread to exit the waiting method and jump straight into the catch block.

package com.springboot.tutorial.deadlock;


import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

class Runnable1 implements Runnable
{

    private Lock l1 = null;
    private Lock l2 = null;

    Runnable1(Lock l1, Lock l2){
        this.l1=l1;
        this.l2=l2;
    }

    @Override
    public void run(){
        String threadName = Thread.currentThread().getName();
        System.out.println(threadName+" going to apply lock l1");
        l1.lock();
        System.out.println("Lock l1 applied by "+threadName);

        try{
            Thread.sleep(3000);

        catch(InterruptedException ex){
            System.out.println(ex.getMessage());
        }

        System.out.println(threadName+" going to apply lock l2");

        System.out.println("Lock l2 applied by "+threadName);
    }
}

- Deadlock Prevention Time Out Code


package com.springboot.tutorial;


import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

class Runnable1 implements Runnable
{

    private Lock l1;
    private Lock l2;

    Runnable1(Lock l1, Lock l2){
        this.l1=l1;
        this.l2=l2;
    }

    @Override
    public void run(){

        String ThreadName = Thread.currentThread().getName();

        while(true){
            int attemptCount = 0;
            while(!tryToDoLocks()){
                attemptCount+=1;
                System.out.println(ThreadName+" unable to apply lock on l1 and l2");
                try {
                    Thread.sleep(7000);
                }
                catch(InterruptedException ex){
                    System.out.println("Thread Sleeping");
                }
            }
            if(attemptCount>0){
                System.out.println(ThreadName+" occupied lock after "+attemptCount+" attempts");
            }

            l1.unlock();
            l2.unlock();
        }
    }

    public boolean tryToDoLocks() {

        String threadName = Thread.currentThread().getName();
        try{
            boolean tryingToLock1 = l1.tryLock(1000, TimeUnit.MICROSECONDS);
            if(!tryingToLock1){
                System.out.println(threadName +" not able to lock l1");
                return false;
            }
        }
        catch(InterruptedException ex){
            System.out.println("Unable to lock l1");
        }

        try{
            boolean tryingToLock2 = l2.tryLock(1000, TimeUnit.MICROSECONDS);
            if(!tryingToLock2){
                System.out.println(threadName +" not able to lock l2");
                l1.unlock();
                return false;
            }
        }
        catch(InterruptedException ex){
            System.out.println("Unable to lock l2");
        }
        return true;
    }
}
